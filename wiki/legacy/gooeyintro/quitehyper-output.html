Enhanced DTs: Boiler Room DTs

Sometimes our code seems a bit messy, or verbose, with repetitive boilerplate, or whatever. After fighting it every which way, we realise there's just no way getting around it. D'oh!

So what to do? One "solution" is to isolate messy code in a class whose main job is... to isolate our messy code! You know, like sweeping it under a rug. So that it doesn't clutter up our important code. If it's pure brainless boilerplate with no logic, then we can look the other way when we have to edit it, and when we don't.

Having piles of anonymous classes, and anonymous classes within anonymous classes, does tend to look a bit messy. Closures would neaten them up somewhat, but we don't have closures yet. In the meantime... let's sweep them under the rug!

Since the above method is running outside the EDT, we need to invoke any GUI updates eg. updateGuiNewMessages(), using invokeAndWait(). Of course we could put this in the process() method in our SwingWorker, and invoke it using publish(). And we can update the GUI in the done() method too.

But let's consider we have a succession of long tasks (eg. communicating with a webservice) with interspersed GUI updates to perform, and let's say we wish to program this relatively naturally without loads of nested anonymous classes.


EDT Support

In the code above, we delegate to an EDT support class, which is implemented as follows. 

public class BasicEdtInvoker {    
    ...
    public void invokeAndWait(final Runnable runnable) {
        try {
            SwingUtilities.invokeAndWait(runnable);
        } catch (InvocationTargetException ite) {
            showExceptionDialog(ite.getCause());
        } catch (InterruptedException ie) {
            showExceptionDialog(ie);
        }
    }
            
    public void startWorker(final Runnable runnable) {
        SwingWorker worker = new SwingWorker() {
            protected Object doInBackground() throws Exception {
                try {
                   runnable.run();
                   return null;
                } catch (Throwable e) {
                   showExceptionDialog(e);
                }
            }
        };
        worker.execute();
    }
    
    public void setEnabled(final Component component, final boolean enabled) {
        invokeAndWait(new Runnable() {
            public void run() {
                component.setEnabled(enabled);
            }
        });
    }    
    ...
    JFrame applicationFrame = application.getApplicationFrame();
    JPanel glassPane = application.getGlassPane();

    public void setWaitCursor(final boolean waitCursor) {
        invokeAndWait(new Runnable() {
            public void run() {
                if (applicationFrame != null) {
                   Cursor cursor = Cursor.getPredefinedCursor(Cursor.WAIT_CURSOR);
                   if (!waitCursor) cursor = null;
                   applicationFrame.setCursor(cursor);                
                   if (glassPane != null) glassPane.setVisible(waitCursor);
                }
            }
        });
    }

    public void showExceptionDialog(Throwable e) {
        JOptionPane.showMessageDialog(null, e.getMessage());
    }    
}

So from our doInBackground() methods, we use the above helper to perform any GUI updates, using invokeAndWait() as necessary.


Isolated Boilerplated Peer

Getting back to our application...

Let's sweep all our messy "closures" in SupportAssistantGui into SupportAssistantGuiInvoker, which we implement as follows. 

public class SupportAssistantGuiInvoker extends BasicEdtInvoker {
    SupportAssistantGui target;
    
    protected SupportAssistantGuiInvoker(SupportAssistantGui target) {
        this.target = target;
    }
    
    protected void checkForNewMessages() {
        startWorker(new Runnable() {
            public void run() {
                target.checkForNewMessages();
            }
        });
    }
    
    protected void updateGuiWithNewMessages(final CheckForNewMessagesResponse response) {
        invokeAndWait(new Runnable() {
            public void run() {
                target.updateGuiWithNewMessages(response);
            }
        });
    }        
    ...
}

So the above is essentially a delegating class, full of boilerplate code, to invoke methods on SupportAssistantGui either in the background, or in the EDT. It's messy, but it's simple, and easy to cut and paste the boilerplate, because it's full of all the same boilerplate code. Like i said, we look the other way when we are working on this prodigal class.

Then our main SupportAssistantGui becomes a bit neater, as follows. 

public class SupportAssistantGui extends JFrame implements ActionListener {    
    SupportAssistantGuiInvoker invoker = new SupportAssistantGuiInvoker(this);
    ...
    public void actionPerformed(ActionEvent event) {
        if (event.getSource() == checkForNewMessagesButton) {
            if (event.getSource() == checkForNewMessagesButton) {
                invoker.checkForNewMessages();
            }
        }
    }
    
    protected void checkForNewMessages() {
        invoker.setEnabled(checkForNewMessagesButton, false);
        try {
            QRequest message = new CheckForNewMessagesRequest();
            QResponse response = connection.sendRequest(message);
            invoker.updateGuiWithNewMessages((CheckForNewMessagesResponse) response);
        } catch (IOException e) {
            invoker.showExceptionDialog(e);
        } finally {
            invoker.setEnabled(checkForNewMessagesButton, true);
        }
    }
    ...
    protected void updateGuiWithNewMessages(CheckForNewMessagesResponse response) {
       ... // update inboxView
    }   
}

I would say that looks more like we would want it to look, what do you think!?


Conclusion
"Mmmm... crumbled-up cookie things." Homer Simpson

Even in our favourite language, there are some things that get quite messy, like Swing EDT programming using anonymous classes nested in anonymous classes, strung together with anonymous classes. Closures would make the code less verbose.

In the meantime, we can always isolate code into a peer class, that is full of boilerplate code, and otherwise contains no logic that unsuspecting eye balls will have to glaze over later. Then our important code is less cluttered and easier to read. More donkey work is required up front for this, but the result is neater code that will be easier to read and maintain later.

In this case, we aren't using any magic tricks, which is good. We are just increasing the number of the classes in our application. It's a logical argument that if you increase the number of artifacts (classes and/or methods) in the application, then the size and complexity of the individual artifacts will tend to decrease. This is my personal silver bullet, if i have one.

Having successfully avoided magic tricks this time, next time we'll start investigating what those magic tricks might be! First we'll try dynamic proxies, to remove the boilerplate code. And finally we'll look at using CGLIB's "Enchancer" to get us some Enhanced DTs.
